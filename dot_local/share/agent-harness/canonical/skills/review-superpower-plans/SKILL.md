# Review Superpower Plans

## Overview

Run an adversarial review before coding starts. Preserve baseline quality checks, then actively challenge assumptions, design ideas, and implementation details to force a stronger final plan.

Announce at start: "I'm using the review-superpower-plans skill for adversarial plan review."

## Inputs

Gather these before reviewing:
- Plan file(s) under review
- Source requirements (ticket/spec/request)
- Repository constraints (`AGENTS.md`, architecture docs, test/build rules)

`Plan file(s) under review` is mandatory. Do not start review without it.

If the user did not specify a plan, pause and request it first with concise wording:
- "Please specify the exact plan to review (path, pasted content, or link)."

For other missing inputs (requirements/constraints), state assumptions explicitly in the review.

## Review Workflow

1. Classify plan type:
- Design plan (architecture/tradeoffs, usually from brainstorming)
- Implementation plan (execution tasks, usually from writing-plans)
- Mixed plan (contains both)

2. Validate plan target is explicit:
- If missing, stop and ask for the specific plan. Do not provide findings yet.

3. Load [review checklist](references/review-checklist.md) and apply relevant sections.

4. Perform three passes:
- Pass 1: correctness and completeness (requirements, constraints, edge cases)
- Pass 2: adversarial challenge (assumptions, design alternatives, failure scenarios)
- Pass 3: executability and verification (sequence, commands, tests, rollback)

5. Produce findings sorted by severity, then decide verdict: `approve` or `needs revision`.

6. Request execution approval before making changes:
- Ask explicitly whether approved edits should be applied now.
- Ask explicitly whether changes should be committed.
- Do not apply edits or create commits without explicit user approval.

## Severity Rules

- `P0` Blocker: plan cannot be safely executed or will likely fail immediately
- `P1` High: major risk of rework, defects, or dead-end execution
- `P2` Medium: meaningful clarity/maintainability/testing gap
- `P3` Low: polish or optional optimization

Do not inflate severity. Keep findings specific and actionable.
Escalate severity when an unproven assumption or weak design choice can cascade into major rework.

## Output Contract

Always return:
1. Findings (highest severity first, each with impact + exact fix direction)
2. Challenged assumptions (what was challenged and why)
3. Stronger alternatives for key design/implementation decisions
4. Open questions/assumptions
5. Verdict (`approve` or `needs revision`)
6. Minimal revision checklist the author can apply
7. Approval prompt: ask whether to apply approved edits and whether to commit them

Use this concise finding format:
- `[P1] Missing rollback path` - Rotation service migration has no backout step; add explicit revert commands and state restoration checks.
- `[P1] Unjustified assumption` - Plan assumes encrypted stream seeks are cheap; add benchmark evidence or redesign buffering strategy.

## Superpower-Specific Quality Gates

When reviewing plans generated by superpower skills, enforce these gates:

- `brainstorming` outputs:
  - Problem framing, options considered, and explicit tradeoff decision
  - Constraints and non-goals captured
  - Risks and validation strategy called out
  - Core assumptions are explicit and challenged with counterexamples

- `writing-plans` outputs:
  - Task-by-task execution with exact files/commands
  - TDD flow is explicit (fail -> implement -> pass)
  - Verification and expected outcomes are measurable
  - Sequence is dependency-safe and includes checkpoint boundaries
  - Implementation details are defended against failure scenarios

- `executing-plans` handoff readiness:
  - Plan is executable without hidden context
  - Risky operations have guardrails
  - Stop conditions and escalation points are explicit
  - Known uncertainty points include fallback decisions or stop gates

If these gates fail, verdict is `needs revision`.
